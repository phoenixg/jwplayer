视频分发: HTTP Pseudo-Streaming / HTTP伪流式视频

目的

此指南讨论并提供了用 Flash v4在JW Player里播放 HTTP Pseudo-streaming / HTTP 伪流式视频的方法.

介绍

MP4 和 FLV 视频都可以通过一个 HTTP伪流式（ pseudostreaming）的机制来播放. 这个机制允许播放还没有下载到的部分. YouTube就是一个应用伪流式的例子. HTTP 伪流式 可以通过设置 option type=http 来启用.

HTTP 伪流式结合了 HTTP 下载的优势 (穿透任何防火墙, 低带宽的用户只要等待下载完成即可播放) ，而且还可以直接播放未下载到的部分. 唯一的HTTP 伪流式相对于flash官方的 RTMP 流 的缺点就是安全性稍差 (HTTP 比 RTMP更容易侦测) ，并且在寻找（seeking）大视频文件时需要更长的加载时间 (>15 分钟).

不要把HTTP 伪流式和HTTP 动态流式相混淆了.后者是目前 Adobe开发的全新机制, 它把原视频切割成一个个所谓的 chunks ，每个几秒. 视频播放器随之把这些 chunks 粘贴在一起进行播放. JW Player (4x) 的版本不支持 HTTP 动态流式.

服务器

HTTP 伪流式不是任何服务器默认就有的. 需要安装一个服务器端的模块来启用它. 以下是两个应用最广泛的 (而且是开源的) 模块:

  • H264 流式模块（用于 Apache, Lighttpd, IIS 和 NginX）.支持 MP4 视频.
  • FLV 流式模块 （用于 Lighttpd）.支持FLV 视频.

有几家 CDN提供商 (内容分发网络) 也支持 HTTP伪流式. 以下几个都经过了测试： Bitgravity, CDNetworks, Edgecast 和 Limelight.

除了使用服务器端的模块外,伪流式还可以通过服务器端的脚本来启用 (比如使用 PHP 或 .NET). 但我们不建议如此做 - 因为这样的脚本很耗系统资源, 而且安全性差,且只能用于 FLV 文件. 其中比较好的伪流式脚本是： Xmoov-PHP.

机制

HTTP 伪流式是这样工作的:

当视频加载时, 播放器读取并存储一系列的 seekpoints 作为视频的metadata（元数据）. 哪里有一个新关键帧（ keyframe）开始，那里就会插入这些seekpoints 进视频 (不仅是秒，还有字节) .随着这些插入,就是向服务器作了一次请求.

当用户寻求（seek）一个未下载到的部分时, 播放器会将它翻译成视频当中最近一处的seekpoints. 然后, 播放器对服务器作一个请求, 就用这个seekpoints作为参数. 对于 FLV 视频来说, 偏移量总是以字节数提供的:

http://www.mywebsite.com/videos/bbb.flv?start=219476905

对于 MP4 视频, 偏移量总是以秒来提供的:

http://www.mywebsite.com/videos/bbb.mp4?starttime=30.4

这时服务器端就会从偏移量起返回视频. 因为该视频的第一个帧是关键帧,播放器就能够正确地加载并播放它. 如果服务器从一个任意的偏移返回视频,播放器就不能取这个流.

注意: 有些 FLV 编码器在编码视频的时候不会加进 seekpoints metadata （seekpoints元数据）. 如果没有这个数据, HTTP 伪流式就不能正常工作.如果你觉得你的视频可能缺少了metadata, 那么可以使用我们的 Metaviewer 插件 来检查视频是否具有正确的metadata. 应该是有seekpoints 或 keyframes 列表显示出来. 如果没有显示的话, 可以使用 FLVMDI tool 来解析你的 FLV视频，并注入metadata.

Startparam起始参数

当播放器使用一个偏移量来请求视频时, 它使用 start 作为偏移量参数的名称（FLV 视频）， 使用starttime 作为偏移量参数的名称（MP4视频）:

http://www.mywebsite.com/videos/bbb.flv?start=219476905
http://www.mywebsite.com/videos/bbb.mp4?starttime=30.4

这些名称被服务器端的模块和CDN提供商广泛使用.然而, 有时候 CDN 可能使用一个不同的名称作为参数. 如果是那样, 那么就使用 http.startparam 选项来设置自定义的偏移量名称. 以下是一些CDN使用不同名称的例子:

  • Bitgravity 使用 http.startparam=apstart（ MP4 视频）.
  • Edgecast 使用 http.startparam=ec_seek （ FLV 视频）.
  • Limelight 使用 http.startparam=fs （ FLV 视频）.

以下是一个SWFObject 嵌入式代码的例子 ， HTTP 伪流式和自定义的起始参数被启用:

<div id='container'>The player will be placed here</div>

<script type="text/javascript">
  swfobject.embedSWF('player.swf','container','480','270','9.0.115','false',{

    file:'http://bitcast-a.bitgravity.com/botr/bbb.mp4',
    type:'http',
    'http.startparam':'apstart'

  });
</script>

播放列表

HTTP 伪流式还可以在播放列表中启用, 这要调整 JWPlayer 命名空间. 可以在播放列表中为每个条目设置一个type 和 http.startparam 选项. 如果是这样的话, 你就无需在嵌入代码中设置了 (只要把 file 指向你的播放列表即可).

下面是个例子, 一个 RSS feed和一个视频文件:

<rss version="2.0" xmlns:jwplayer="http://developer.longtailvideo.com/">
  <channel>
    <title>Playlist with HTTP Pseudostreaming</title>

    <item>
      <title>Big Buck Bunny</title>
      <description>Big Buck Bunny is a short animated film by the Blender Institute,
         part of the Blender Foundation.</description>
      <enclosure url="http://myserver.com/botr/bbb.mp4" type="video/mp4" length="3192846" />

      <jwplayer:type>http</jwplayer:type>
      <jwplayer:http.startparam>apstart</jwplayer:http.startparam>

    </item>
  </channel>

</rss>

与其附加（enclosure）元素t, 你还可以使用 media:content 或 jwplayer:file 元素. 更多请见 XML 播放列表支持.

注意: 不要忘记在 feed 的顶部的 xmlns .播放器需要它 (别的feed阅读器也需要它) 来理解 jwplayer 的各元素.

比特率切换

像 RTMP 流一样, HTTP 伪流式包含为每个观看者优化每个视频的能力 . 我们称这一个机制为 比特率切换.

要使用比特率切换, 你需要几个版本的 MP4 或 FLV 视频, 每个都是不同的质量 (不同的分辨率和比特率). 这些不同的视频，使用一个 mRSS 播放列表来进行加载 (见下面的例子). 播放器会认出你的视频属于何种的 levels ，然后自动选择最高质量的那个:

  • 适合服务器的带宽（ bandwidth ）» 客户端的网络连接.
  • 适合播放器的显示宽度（ width ） (或者, 说得更精确些, 不比 20% 更大).

观看者持续观看视频, 播放器会反复检查这一决定 (并且有可能切换)，随着下列事件的发生进行变化:

  • startup, 视频加载时马上开始第一次计算带宽情况.
  • fullscreen 开关, width 会急剧地改变.比如, 浏览器全屏，用户又有很好的网络, 播放器就会提供 HD 版本的视频.
  • 每次seek视频时. 播放器重新缓冲视频流 .这时也会监测带宽是否发生了改变.

注意: 如果带宽发生突变，引起视频重新缓冲，那么播放器不会进行带宽切换, 实际上，我们发现如果不这样做，会发生不断的切换，影响用户体验.（没错）

例子

下面是几个播放列表比特率切换的例子 (只放了一个项目). 注意这和常规的（ regular） HTTP 伪流式播放列表类似, 除了每个项目上放了许多个视频这个不同外. mRSS扩展是唯一一种提供那么多元素的方法，包括 bitrate 和 width 属性:

<rss version="2.0" xmlns:media="http://search.yahoo.com/mrss/"
  xmlns:jwplayer="http://developer.longtailvideo.com/">
  <channel>
    <title>Playlist with HTTP Bitrate Switching</title>

    <item>
      <title>Big Buck Bunny</title>
      <description>Big Buck Bunny is a short animated film by the Blender Institute,
         part of the Blender Foundation.</description>
      <media:group>

        <media:content bitrate="1800" url="http://myserver.com/bbb-486.mp4"  width="1280" />
        <media:content bitrate="1100" url="http://myserver.com/bbb-485.mp4" width="720"/>

        <media:content bitrate="700" url="http://myserver.com/bbb-484.mp4" width="480" />
        <media:content bitrate="400" url="http://myserver.com/bbb-483.mp4" width="320" />

      </media:group>
      <jwplayer:type>http</jwplayer:type>
      <jwplayer:http.startparam>apstart</jwplayer:http.startparam>
    </item>

  </channel>
</rss>

一些提示:

  • bitrate 属性必须用 kbps 单位, 这是定义在 mRSS spec里的规定. width 属性的单位必须是像素.
  • 推荐根据质量对视频进行排序, 质量最好的排在最前面. 大多数 RSS阅读器会选择这个进行播放.
  • 这个 feed里面的4个不同级别的视频，实际上是我们推荐进行比特率切换的宽屏 MP4 视频. 对于4:3 视频或 FLV 视频, 你可能想要增加比特率或稍稍减少一下分辨率.
  • 有些内容发布商进行比特率切换时只是改变了下比特率. 播放器可以这么播放, 但是对比特率和分辨率都做修改的话，级别选项就更多一些 (视频的重复利用, 比如把最小的提供给手机用播放).
  • 这里的media:group 元素是可选的,但它在形式上整理组织了一下，看起来更有条理.
